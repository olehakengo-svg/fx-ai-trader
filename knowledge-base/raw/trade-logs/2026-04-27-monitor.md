# Trade Monitor Alerts: 2026-04-27

## Alert: 01:25 UTC

### 検出された問題
- 🚨 STATUS API取得失敗: The read operation timed out
- ⚠️ **直近4時間トレード0件** かつ全モードWAIT — 発火条件を満たしていないか、フィルターで全遮断の可能性

### 診断
## 原因分析

### 推測（確認前）

**STATUS API タイムアウトについて**
- APIサーバープロセスがハング・フリーズしている可能性
- サーバーリソース枯渇（メモリリーク、CPU過負荷、ディスク満杯）によるレスポンス不能
- ネットワーク層の障害（ロードバランサー、プロキシ、ファイアウォールのセッション詰まり）
- Pythonプロセスがデッドロック状態に陥っている可能性

**トレード0件・全モードWAITについて**
- APIが死んでいるためステータス取得自体が不正確（偽陽性の可能性大）
- ただし仮にプロセス自体は生きていても、フィルター（スプレッド・ボラティリティ・時間帯）が全条件を遮断している可能性
- ブローカーへの接続（MT4/MT5またはREST）が切断されたままになっている可能性

---

### 確認事項（優先順）

**サーバー死活確認（最優先）**
- [ ] サーバーへSSH直接ログインできるか
- [ ] `top` / `htop` でCPU・メモリ・ロードアベレージ確認
- [ ] `df -h` でディスク残量確認
- [ ] 対象プロセスが存在するか（`ps aux | grep [プロセス名]`）

**プロセス・ログ確認**
- [ ] アプリケーションログファイルを直接確認（ログ`[]`はAPI経由取得失敗のため実ファイルを見る）
- [ ] システムログ（`/var/log/syslog` 等）にOOM killerやパニックの痕跡がないか
- [ ] プロセスがゾンビ化していないか

**ネットワーク・接続確認**
- [ ] ブローカーAPIへの疎通確認（`ping` / `curl`）
- [ ] APIサーバーのポートがLISTENしているか（`ss -tlnp`）
- [ ] タイムアウト値の設定が短すぎないか（暫定的な問題の場合）

---

### 対処方針

1. **まずサーバーに直接SSHして生死確認** — 監視系がAPIに依存しているため、監視ツール側の結果を盲信しないこと
2. リソース枯渇が確認された場合 → 不要プロセス停止、ディスク掃除後にサービス再起動
3. プロセスがハング確認された場合 → `kill -9` 後に再起動、再起動後の復帰ログを必ず

## 原因分析

### 【推測】含み益がShadowに留まりLIVEに昇格しない根本原因

**1. TREND_BULL TF Bypassによる強制Shadow化**
- `post_news_vol×EUR_USD`が `TREND_BULL WR=0% → shadow` でShadow送りされている
- `daytrade_gbpusd:regime_trend_bull_dt_tf` のブロックカウントが67と最多クラス
- EUR_USD / GBP_USD共に「上位TFのTREND_BULL判定」がLIVE昇格を阻んでいると推測

- `daytrade:score_gate` 136件、`daytrade_gbpjpy:score_gate` 79件
- USD_JPYの`streak_reversal`がscore=-3.00で連続ブロック → daytrade系全般でLIVEシグナルが出ていない状態

**3. `PAIR_PROMOTED`未設定**
- Shadow含み益が+49.8pipに達しているにもかかわらず、当該ペアがPAIR_PROMOTEDリストに入っていないため、自動昇格ロジックが発火しない

---

### 【確認事項】対処前に確認すべきこと

- [ ] `TREND_BULL WR=0%` の根拠データが現在も正しいか（上位TF足の取得・計算が正常か）
- [ ] `regime_trend_bull_dt_tf` の判定閾値設定値と、現在のEUR_USD / GBP_USDの実際のレジーム状態が乖離していないか
- [ ] PAIR_PROMOTEDの昇格条件（pip閾値・継続時間）の現在設定値
- [ ] `main_loop_restarts=1` の再起動タイミングとShadowポジション生成タイミングの前後関係（再起動後にLIVEステートが初期化されていないか）
- [ ] `score=-3.00`（USD_JPY daytrade）のスコア算出元データ異常の有無（`unknown`ラベルのブロックも散見されるため、スコアソースのNaN/未初期化を疑う）

---

## 対処方針

### 即時対応

- **`post_news_vol×EUR_USD` および `sr_break_retest×GBP_USD` をPAIR_PROMOTEDに追加**し、Shadow→LIVE昇格を手動でアンロック
- `TREND_BULL WR=0%`の上位TFデータを手動取得し、計算結果が実態と一致しているか突合確認

- `regime_trend_bull_dt_tf` ブロックロ

## Alert: 15:28 UTC

### 検出された問題
- ⚠️ **block_counts異常** — rnb_usdjpy:direction_filter=207

### 診断
## 原因分析

### 【推測】rnb_usdjpy:direction_filter=207 の原因候補

- **相場の一方向継続（トレンド固着）**
  USDJPYが強いトレンド（上昇または下落）を継続しており、RnBストラテジーが許容する方向と逆のシグナルが連続発生し、フィルターが大量にブロックし続けている可能性
- **direction_filterの閾値設定ミス or ロジック異常**
  フィルター判定ロジックが特定の相場状態（例：WIDE_RANGE/RANGE_TIGHT混在）でほぼ全シグナルを棄却するバグ・設定値の劣化が発生している可能性
- **MTFバイアスの固着**
  ログに `mtf=range_tight d1=0 h4=1 vol=squeeze` が繰り返し出現しており、上位足のバイアスが固定されたままdirection_filterが解除されない状態に陥っている可能性
- **カウンターのリセット漏れ**
  block_countsが起動からの累積カウントであれば、main_loop_restarts=1（再起動歴あり）にもかかわらず当該カウンターだけリセットされていないバグの可能性

---

### 【確認事項】優先度順

1. **rnb_usdjpy の direction_filter 判定ソースを確認**
   - 現在どの方向（BUY/SELL）をフィルターしているか
   - フィルター判定に使用しているMTF・D1・H4の現在値を取得
   - フィルター解除条件が満たされていない理由を特定

2. **block_countsのカウント仕様を確認**
   - 累積カウントか／一定期間のウィンドウカウントか
   - main_loop_restart時にリセットされる設計かどうか
   - 他ストラテジーの最大値（daytrade_eurgbp:score_gate=106）と比較して207が突出しているかを評価

3. **USDJPY現在の相場状態を確認**
   - 実際にトレンド継続中か、レンジ入りしているかをチャートで確認
   - rnb戦略が本来エントリーすべき方向と現在の価格アクションの整合性を確認

4. **rnb_usdjpy のシグナル生成ログを個別に抽出**
   - DIRECTION_FILTERのブロック頻度（時間帯分布）を確認し、特定時刻から急増していないか確認

---

## 対処方針

- **即時対応（低リスク）**
  - rnb

### 診断
## 原因分析

### 推測（ログ・カウントから推定）

**① rnb_usdjpy:direction_filter=318（最多）**
- USDJPYが方向性フィルターで一方向（おそらくSELL方向）に継続ブロックされている
- MTFログで `range_tight / vol=squeeze / d1=0` が確認されており、**レンジ・ボラティリティ収縮局面**でrnbストラテジーのシグナルが方向性判定と常に不一致になっている可能性が高い
- ループ間隔が短い（interval設定）ままシグナル生成頻度が高く、同一判定でカウントが積み上がっている構造的な積み増しの可能性

**② daytrade_eurjpy:score_gate=268（次点）**
- ログに `unknown score=-0.69〜-0.70 < 0 | WAIT EUR_JPY` が頻出
- スコア算出のentry_typeが `unknown` のまま評価されており、**entry_typeの分類ロジックが想定外の状態を返している**可能性
- WAIT状態でもscore_gateチェックが毎ループ走っており、実質的に無限ブロックループになっている可能性

---

### 確認事項（実機・コードで要検証）

**rnb_usdjpy:direction_filter について**
- [ ] direction_filterの判定条件（閾値・参照するMTF足）が現在の相場レンジ環境に対して過剰に厳しくなっていないか
- [ ] シグナル生成→フィルター評価のループ間隔が意図した設定値か（interval=30が実態と一致しているか）
- [ ] `d1=0` が続いている期間の長さ（日足レベルの方向性が長期間中立の状態ではないか）

**daytrade_eurjpy:score_gate について**
- [ ] entry_typeが `unknown` を返すケースのハンドリングがscore計算ロジックに実装されているか
- [ ] WAIT状態のシグナルに対してscore_gateが毎ループ評価されるべき設計か（本来はスキップすべきか）
- [ ] score=-0.69/-0.70の値が固定的かどうか（毎回ほぼ同値 → 入力データが更新されずキャッシュ参照になっていないか）

---

## 対処方針

- **即時対応**
  - rnb_usdjpy のdirection_filterブロック中はシグナル生成自体を一時停止し、カウント積み増しを抑制する（フィルター拒否後の再評価にクールタイムを設ける）
  - daytrade_eurj
