# KB Drift on Context Limit — コード変更とKB更新の乖離

**発見日**: 2026-04-13
**修正**: 手動整合（git log全件精査→4ファイル更新）

## 何が起きたか
v8.5でObsidian KB構築を完了。その後v8.6→v8.7→v8.8とコード変更を次々コミットしたが、使用量制限でセッションが途切れ、KBへの反映が一切行われなかった。

## 症状
- changelog.md: v8.4で停止（v8.5-v8.8の4バージョン分が欠落）
- wiki/index.md: "v8.4, 2026-04-12" のまま
- session log: Phase 9で終了（Phase 10-12が欠落）
- edge-pipeline.md: 8戦略がStage 1のまま（実際はStage 4-6に昇格済み）
- 未解決事項: 「DSR未実装」「GBPアジア除外未実装」（両方v8.6で実装済み）

## 根本原因
コード変更とKB更新を別コミットにしていた。セッションが途中で切れると、コードは残るがKBは古いまま固定化する。

## 影響
- 次セッション開始時にKBを読んでも実態と合わない情報で判断することになる
- 「未実装」と記録された機能を再実装しようとする無駄が発生しうる
- バージョンの不一致で、どのデータがどの条件で取得されたか不明になる

## 対策

### ルール1: feat()コミットにKB更新を同梱
コード変更のコミットに、関連するchangelog/wiki/session logの更新も含める。分離しない。

### ルール2: セッション終了が近い場合はKB優先
使用量制限が近づいた場合、新機能の実装よりKBの整合性更新を優先する。コードは次セッションで書けるが、コンテキストは失われる。

### ルール3: セッション開始時にgit log vs changelog整合チェック
次セッション開始時に `git log --oneline -10` と changelog.md の最新エントリを比較し、乖離があれば最初に修正する。
