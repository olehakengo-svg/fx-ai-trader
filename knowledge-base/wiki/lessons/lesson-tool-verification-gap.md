# Lesson: ツール検証ギャップ — 作ったツールを使わずに判断した

## 発生日
2026-04-13

## 何が起きたか
BT/本番乖離トラッキング関数 (preprocess_bt_divergence) を実装し、daily_report.pyに組み込んだ。
しかし:
1. スモークテストが「クラッシュしない」しか検証しなかった（空結果=OKと判定）
2. 実BTファイルでの正例テストを行わなかった
3. パーサーのregexバグで0件出力（BTファイルのペアヘッダー形式と不一致）
4. バグに気付かないまま次のタスク（session_time_biasスコア変更等）に進んだ
5. 戦略変更時にBT乖離ツールの出力を参照しなかった — ツールの存在自体を忘れた

## 根本原因（5層）
1. **スモークテストの不十分さ**: 「壊れない」≠「正しい」。正例テスト(非空出力)が必要
2. **ツール作成と使用の分離**: 作る→忘れる→判断は直感で。統合フローがない
3. **並列タスク実行による検証スキップ**: 5タスク高速実行で各ステップの検証が省略
4. **クオンツファーストのメタ適用漏れ**: 分析ツール自体に「分析→判断→実装」を適用しなかった
5. **成功基準の未定義**: 「何件抽出されれば正常か」が定まっていなかった

## 再発防止
- 分析関数は実データで正例テスト必須（TestBtDivergenceParser追加済み）
- 作ったツールは即座に実データで使用 → 出力を引用してから判断
- 空結果はバグとして疑う（「データがない」で片付けない）
- [[claude-harness-design]] にRC6対策ルール追加済み
