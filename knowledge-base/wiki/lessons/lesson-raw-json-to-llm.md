# Lesson: 生JSONをLLMに丸投げしない

**日付**: 2026-04-13
**カテゴリ**: データパイプライン設計
**深刻度**: Critical（レポート品質が根本的に破壊される）

## 何が起きたか

daily_report.pyがtrades API（500件）の生JSONを `[:5000]` で文字列切断してClaude APIに送信。
299件中4件しかLLMに届かず、アナリストは「JSONが途中切断」と報告。
レポート品質がゴミになった。

## 根本原因

「LLMは賢いから生データを渡せば集計してくれる」という設計思想。
実際にはトークン上限・文字列切断・計算精度の問題で機能しない。

## 教訓

### 原則: LLMに渡すのは「判断材料」であって「生データ」ではない

1. **前処理はPython側で行う** — 集計・フィルタリング・突合はコードで確実に
2. **LLMには集計済みテーブルと文脈を渡す** — 「この数字が何を意味するか判断せよ」
3. **文字列切断 `[:N]` はデータ破壊** — JSON構造が壊れ、末尾のデータが消える
4. **セッション時間帯のフィルタもコード側** — LLMにopen_timeのパースを任せない
5. **二重データソース（デモ/OANDA）の突合もコード側** — LLMに2つのAPIレスポンスをマージさせない

### 適用タイミング

- LLM APIにデータを送る全てのパイプライン（daily_report, weekly_audit, trade_monitor）
- 新しいレポート系スクリプトを作るとき
- Claude APIのuser messageを構築するとき
